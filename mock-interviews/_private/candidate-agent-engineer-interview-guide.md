# AI Agent 工程师面试提纲

> 日期：2026-07-16
> 用途：第一次面试时的面试官辅助提纲
> 面试重点：Agent 工程、SDD、Agentic SDLC、AI 治理、多 Agent 协作
> 面试风格：少问实现细节，多问工作方法、工程判断和真实经历

## 1. 候选人画像

根据候选人简历，候选人具备以下背景：

- 9 年 Java 后端开发经验，熟悉 Spring / Spring Boot / Spring Cloud。
- 有 MySQL、PostgreSQL、Redis、ES、RabbitMQ、Kafka、Canal、Docker、Kubernetes 等工程经验。
- 接触过 Spring AI Alibaba、Dify、Prompt、Tool Calling、RAG、长期记忆和上下文管理。
- 参与过 AI 医生智能体平台、AI-MDT 健康管理系统、OCR + LLM 体检报告解析等项目。
- 简历提到 AI 问答日均 2 万次、WebSocket / SSE 流式输出、会员支付和邀请积分体系。
- 有项目管理经验，曾带 4 人团队。

### 面试定位

这不是一场模型算法或 Java 细节面试。主要需要判断候选人是否能够：

1. 把 Agent 当作一个可交付、可观测、可治理的工程系统。
2. 在需求不清晰时建立 Spec、边界和验收标准。
3. 设计 Agent 参与研发的完整生命周期。
4. 合理拆分多 Agent，并处理协作、冲突和失败。
5. 在效率、自主性、安全、成本和责任之间做判断。

## 2. 面试官的基本原则

不要先问：“你了解 SDD、Agentic SDLC、AI Governance 吗？”候选人可能没有使用过这些术语，但实际做过类似工作。

更好的问法是给场景：

> 如果让 Agent 参与一个真实项目，你会如何约束它、验收它，并保证出了问题可以追溯？

面试时重点追问三件事：

- **你具体负责什么？** 区分个人贡献和团队成果。
- **遇到失败怎么办？** 比只听成功案例更能判断经验深度。
- **如何验证做对了？** 判断他是否有验收、评估和治理意识。

不要因为候选人不会某个术语就直接扣分。重点看他能否讲清楚目标、边界、决策、协作和结果。

## 3. 建议面试流程：45 分钟

| 时间 | 模块 | 目标 |
|---|---|---|
| 0—5 分钟 | 项目经历 | 确认个人贡献和最熟悉的项目 |
| 5—15 分钟 | Agent 工程方法 | 判断是否理解 Agent 工程的核心问题 |
| 15—25 分钟 | SDD / Agentic SDLC | 判断是否能建立结构化交付流程 |
| 25—37 分钟 | 多 Agent 协作 | 判断拆分、通信、冲突和失败处理能力 |
| 37—43 分钟 | AI 治理 | 判断权限、安全、审计和责任意识 |
| 43—45 分钟 | 总结与候选人提问 | 观察表达、优先级和价值观 |

## 4. 开场问题：先确认项目真实性

### 问题

> 请选择一个你最熟悉的 AI 项目，重点讲三件事：你个人负责什么、最难的问题是什么、最后如何判断项目做成功了。

### 追问

- 哪些部分是你亲自设计或实现的？
- 哪些部分由其他同事负责？
- 项目中有过什么失败或返工？
- 如果今天重新做，你会改哪三件事？
- 项目成功的指标是什么？是业务指标、质量指标，还是工程指标？

### 观察点

加分信号：能够讲出具体决策、限制条件、失败经验和前后变化。

风险信号：只讲“我们团队”，大量复述技术名词，说不出自己的取舍、结果和问题。

## 5. Agent 工程能力

### 问题 1：Agent 工程师和普通后端工程师最大的区别是什么？

### 追问

- Agent 工程师的主要产物是什么？
- 如何定义 Agent 的职责边界？
- Agent 出错和普通程序出错有什么不同？
- Agent 的“任务完成”应该如何定义？

### 重点观察

比较成熟的回答应该包含：

- Agent 不只是聊天接口或 Prompt。
- 需要设计任务边界、工具边界、状态和决策流程。
- 需要评估、观测、回放和持续改进。
- 不能只看 Demo 能否跑通，还要看失败时是否可控。

### 问题 2：什么情况下应该使用 Agent，而不是普通工作流？

### 追问

- 什么情况下用固定流程更好？
- Agent 的自主决策带来了什么收益？
- 自主性增加后，成本和风险如何控制？

重点观察候选人是否知道：不是所有问题都应该 Agent 化。流程稳定、规则明确、错误代价高的任务，可能更适合确定性工作流。

## 6. SDD 与 Agentic SDLC

这里的 SDD 指 Spec-Driven Development，即先明确需求、边界、验收标准和设计，再让 Agent 执行。一个常见流程是：

```text
Spec → Plan → Tasks → Implement → Validate → Review → Release → Feedback
```

GitHub Spec Kit 官方也将核心流程描述为 `Spec → Plan → Tasks → Implement`：
[GitHub Spec Kit](https://github.github.com/spec-kit/index.html)

### 问题 3：如果产品经理只说“做一个智能客服”，你会要求补充什么？

### 期待候选人关注

- 服务哪些用户？
- 解决哪些问题？
- 哪些问题明确不回答？
- 是否允许调用订单、退款、优惠券等工具？
- 什么时候转人工？
- 什么叫回答正确？
- 如何衡量上线效果？
- 哪些数据和权限可以被 Agent 使用？

### 问题 4：你会如何从需求开始推进一个 Agent 项目？

### 追问

- Spec 至少应该包含哪些内容？
- 如何描述 Agent 的边界和禁止事项？
- 如何把需求拆成可执行任务？
- 需求变化后，如何同步 Spec、设计、任务和测试？
- 如果 Spec 和代码实现不一致，谁应该被认为是正确的？

### 重点观察

好的回答不一定使用 “SDD” 这个术语，但应该体现：

1. 先澄清目标和边界。
2. 定义可验证的验收标准。
3. 再做方案和任务拆解。
4. 让实现、测试和文档保持一致。
5. 需求变化时，更新正式记录，而不是只在聊天里口头说明。

### 问题 5：如果让 Agent 参与需求分析、设计、编码、测试和发布，你会如何设计流程？

可以请候选人画出流程，重点观察是否包含：

```text
需求澄清
  ↓
Spec / Acceptance Criteria
  ↓
方案与风险分析
  ↓
任务拆解
  ↓
实现
  ↓
测试与审查
  ↓
安全和合规检查
  ↓
人工批准
  ↓
发布与线上观测
  ↓
反馈回流
```

### 继续追问

- 哪些环节 Agent 可以自动做？
- 哪些环节必须人工批准？
- Agent 能不能自己合并代码或发布？
- 什么情况下 Agent 必须暂停并请求人类决策？
- 一个 Agent 失败后，是重试、换 Agent，还是转人工？

Agentic SDLC 的重点不是让 Agent 取代所有人，而是把决策、执行、质量门禁和人工审批安排清楚。可参考：[AI-SDLC Framework](https://github.com/ai-sdlc-framework/ai-sdlc)。

## 7. 多 Agent 协作能力

多 Agent 的核心不是“会不会使用某个框架”，而是能否合理拆分职责、定义通信契约、控制权限、处理冲突并定位失败。

### 问题 6：什么情况下应该使用多 Agent，而不是单 Agent？

### 追问

- 什么任务适合拆成多个 Agent？
- 什么任务拆分后反而更复杂？
- 多 Agent 的收益如何衡量？
- 如果单 Agent 已经能完成，为什么还要拆？

### 重点观察

比较好的回答应该提到：

- 任务可以自然拆成不同专业职责。
- 不同阶段需要不同上下文或权限。
- 某些工作可以并行处理。
- 需要独立审查或交叉验证。
- 多 Agent 会增加通信成本、延迟、调度复杂度和错误传播。

### 问题 7：设计一个多 Agent 的 AI 健康助手

场景：系统包含问诊、健康档案查询、医学知识检索、报告解读和安全审核。请候选人自己决定是否拆分 Agent，并说明原因。

可能的结构是：

```text
用户请求
  ↓
任务路由 Agent
  ├─ 健康档案 Agent
  ├─ 医学知识检索 Agent
  ├─ 检查报告解读 Agent
  └─ 健康建议 Agent
          ↓
安全审核 / 事实核验 Agent
          ↓
最终回答或转人工
```

### 追问

- 哪个 Agent 可以调用用户健康档案？
- 哪个 Agent 可以调用外部工具？
- 谁负责最终回答？
- 谁负责安全审核？
- Agent 之间传递什么内容？
- 是否让每个 Agent 都看到完整对话？
- 如果两个 Agent 的结论冲突怎么办？

### 观察重点

- 职责是否清晰。
- 是否遵循最小权限原则。
- 是否避免所有 Agent 都能调用所有工具。
- 是否保留最终审核和人工升级机制。
- 是否考虑多 Agent 带来的额外成本和延迟。

### 问题 8：多 Agent 之间如何传递信息？

### 追问

- Agent 之间传递自然语言，还是结构化结果？
- 如何定义输入输出协议？
- 如何避免上下文越来越长？
- 如何处理重复信息？
- 如果一个 Agent 输出格式不符合要求怎么办？

### 重点观察

优秀回答通常会提到：

- 明确的输入输出 Schema。
- 中间结果结构化。
- 任务状态和上下文分离。
- 只传递必要信息。
- 每一步有状态、超时和错误处理。
- 重要决策保留理由和证据。

可以继续问：

> 如果检索 Agent 返回一段自然语言，而回答 Agent 无法判断哪些是事实、哪些是推断，你会怎么改？

好的方向是让检索 Agent 返回带证据、来源和未解决问题的结构化结果，而不是让下一个 Agent继续猜测。

### 问题 9：多个 Agent 结论冲突时，谁说了算？

场景：一个 Agent 认为用户只是普通健康咨询，另一个 Agent 认为可能存在高风险症状，需要转人工。

### 追问

- 是否简单投票？
- 是否设置 Agent 优先级？
- 哪类 Agent 具有否决权？
- 如何避免多数 Agent 一起犯错？
- 什么情况下必须转人工？
- 冲突结果是否进入日志和评估集？

### 重点观察

成熟的回答应该体现：

- 高风险安全 Agent 可以拥有否决权。
- 不同 Agent 不能简单多数投票。
- 应根据风险等级、证据质量和职责权威性处理冲突。
- 无法解决时采取保守策略或转人工。
- 冲突案例应该被记录并用于后续评估。

### 问题 10：最终答案错了，如何定位是哪个 Agent 出的问题？

期待候选人能够建立完整的运行轨迹：

- 每个 Agent 的输入和输出。
- Tool 调用记录。
- Prompt、模型和版本。
- 每一步耗时和失败原因。
- Agent 之间传递的中间结果。
- 最终答案和用户反馈。
- 一次完整任务的可回放能力。

还应能够区分：

- 路由错误。
- 检索错误。
- Tool 调用错误。
- 某个 Agent 的判断错误。
- 汇总 Agent 的整合错误。

## 8. AI 治理与风险控制

### 问题 11：如果公司开始大量使用 Agent，AI 治理至少要覆盖哪些方面？

期待候选人覆盖：

- 数据权限和隐私。
- 敏感信息脱敏。
- Prompt、知识库和模型版本管理。
- Tool 调用权限。
- 高风险操作审批。
- 输出安全和内容审核。
- 成本与资源控制。
- 模型供应商风险。
- 日志、审计和责任追踪。
- 线上监控和问题响应。

### 问题 12：如果 Agent 可以调用退款、删除数据、发送邮件等工具，你会如何治理？

### 期待回答

- 区分只读操作和写操作。
- 高风险操作需要人工确认。
- 参数和权限不能只依赖模型判断。
- 用户身份和业务权限必须由系统验证。
- 工具使用白名单。
- 限制调用范围、次数和递归深度。
- 保留审计日志。
- 重要操作支持撤销、补偿或回滚。

### 问题 13：Agent 越自主越好吗？

可以让候选人按风险分级回答：

| 风险等级 | 示例 | 建议控制方式 |
|---|---|---|
| 低风险 | 总结文档、分类问题 | 可自动执行 |
| 中风险 | 修改代码、更新知识库 | 自动执行，但需要 Review |
| 高风险 | 退款、删除数据、发布内容 | 必须人工确认 |
| 极高风险 | 医疗诊断、金融决策、权限变更 | 不应完全自动化 |

核心判断不是 Agent 能不能做，而是：出错代价多大、是否可逆、是否能被发现、是否涉及法律或业务责任。

## 9. 上线后的评估与改进

### 问题 14：Agent 上线后用户反馈不好，你会如何定位和改进？

期待候选人建立以下闭环：

```text
线上请求与结果
  ↓
Trace / 日志 / 用户反馈
  ↓
问题分类
  ↓
加入评估集
  ↓
修改 Prompt、Workflow、Tool 或模型
  ↓
离线评估
  ↓
灰度发布
  ↓
线上观察
```

### 追问

- 如何判断是任务定义问题、Prompt 问题还是 Agent 协作问题？
- 如何防止修复一个问题又引入另一个问题？
- 如何建立回归测试集？
- 如何衡量准确性、成本、延迟和用户满意度？
- 如何处理线上出现的新类型问题？

可以关注候选人是否知道：Agent 评估不能只看最终答案，还要看中间行为、工具调用、失败路径和整体运行轨迹。参考：[agentevals](https://github.com/agentevals-dev/agentevals)。

## 10. 最后一道综合题

> 如果让你在公司从零建立一套 Agent 工程规范，前三个月你会先做哪几件事？

观察候选人的优先级。比较合理的方向可能包括：

1. 先选一个明确、低风险、可衡量的试点场景。
2. 建立 Agent 设计模板，定义职责、输入、输出、工具和边界。
3. 建立 Spec、评审、测试和发布流程。
4. 建立基础日志、Trace、成本和质量指标。
5. 为高风险工具建立权限和人工审批机制。
6. 建立评估集和问题回归机制。
7. 再逐步推广多 Agent 和更高自治度。

风险信号是：一上来就采购平台、堆十几个 Agent、追求全自动化，却没有具体业务目标、验收标准和治理机制。

## 12. 最终建议重点问的 6 道题

如果时间有限，直接问下面六道：

1. 什么情况下应该使用多 Agent，而不是单 Agent？
2. 设计一个包含问诊、知识检索、报告解读和安全审核的多 Agent 系统。
3. 多个 Agent 之间如何定义输入输出、共享上下文和权限？
4. 两个 Agent 的结论冲突时，谁说了算？
5. 如果让 Agent 参与软件研发，你会如何设计角色、流程和人工审批点？
6. 如果让你从零建立一套 Agent 工程规范，前三个月你会先做什么？

这六道题可以同时覆盖 Agent 工程、SDD、Agentic SDLC、多 Agent 协作和 AI 治理。

## 13. 面试结束后的核心判断

最重要的不是候选人是否知道某个框架，而是他是否形成了以下工作方式：

> 先定义问题和边界，再让 Agent 执行；
> 用可验证的标准验收；
> 对高风险动作保留人工控制；
> 让 Agent 之间通过清晰契约协作；
> 用日志、评估和线上反馈持续改进系统。

尤其警惕以下情况：

- 把 Agent 等同于 Prompt + Tool Calling。
- 把“回答看起来合理”当成系统成功。
- 只讲效率，不讲责任、权限和风险。
- 认为所有流程都应该自动化。
- 只会堆多 Agent，却说不清每个 Agent 为什么存在。
- 说不清 Agent 冲突时谁负责、失败后如何回放和修复。
