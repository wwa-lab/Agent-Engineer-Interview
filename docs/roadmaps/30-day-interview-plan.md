# 30-Day Interview Plan

This plan assumes 60–90 minutes a day, five days a week. On a shorter schedule, keep the build and explanation days and combine reading days.

## Week 1 — foundations and boundaries

| Day | Focus | Output |
| --- | --- | --- |
| 1 | Skill assessment and role target | Baseline score and target job description. |
| 2 | Tokens, context, embeddings | Two-minute explanation and one context-budget example. |
| 3 | Structured output and tool calling | A schema plus validation failure cases. |
| 4 | What is an agent? | A bounded loop diagram. |
| 5 | Workflow vs agent | A decision table for one real task. |

## Week 2 — retrieval, memory, and protocols

| Day | Focus | Output |
| --- | --- | --- |
| 6 | RAG and chunking | Retrieval assumptions and a no-answer rule. |
| 7 | Retrieval and reranking | A small offline experiment with a quality metric. |
| 8 | Session and long-term memory | Retention, deletion, and privacy boundaries. |
| 9 | Planning and reflection | One case where planning is harmful or unnecessary. |
| 10 | MCP and tool protocols | Explain capability discovery, invocation, and trust. |

## Week 3 — production judgment

| Day | Focus | Output |
| --- | --- | --- |
| 11 | Evaluation design | Five test cases and a scoring rubric. |
| 12 | Observability and tracing | Trace fields for one agent run. |
| 13 | Latency and cost | A rough token/tool latency budget. |
| 14 | Reliability | Retry, timeout, fallback, and idempotency plan. |
| 15 | Security and prompt injection | Threat model and authorization boundary. |

## Week 4 — prove and communicate

| Day | Focus | Output |
| --- | --- | --- |
| 16–18 | Complete one coding challenge | Tests, README notes, and a short demo script. |
| 19–20 | System design case | One blank-page design with two trade-offs. |
| 21–23 | Resume and portfolio | Two quantified bullets and a project README. |
| 24–25 | Behavioral stories | Five stories using the [story structure](../job-search/project-storytelling.md). |
| 26–27 | Timed question banks | Ten answers with follow-up notes. |
| 28 | Senior mock interview | Record one 45-minute run. |
| 29 | Review weak spots | Re-run failing tests and update the study table. |
| 30 | Final rehearsal | Two-minute introduction, system design, coding, and questions for the interviewer. |

The plan is a forcing function, not a scorecard. Keep a visible list of assumptions you still cannot defend.
