# 通用 Agent 工程师面试题库与参考答案

> 版本：v1.0
> 日期：2026-07-16
> 用途：未来多场 Agent 工程师面试的通用题库
> 适用对象：Agent Engineer、LLM Application Engineer、AI Workflow Engineer、AI Platform Engineer
> 面试风格：优先考察工程判断、交付能力、协作能力和治理意识，不把面试变成模型算法考试

## 1. 如何使用这份题库

不要一次问完所有问题。一次 45—60 分钟的面试，建议选择：

- 2 道项目真实性问题
- 3—4 道核心能力问题
- 1 道多 Agent 或系统设计题
- 1 道治理或失败处理问题

对于有经验的候选人，少问定义，多问场景：

> 你遇到过什么问题？当时有哪些选择？为什么这样做？最后怎么验证？

如果候选人没有使用过某个术语，不要仅凭术语判断能力。重点看他的实践是否体现了这个概念。

## 2. Agent 工程师应该具备什么能力

Agent 工程师不是只会调用 LLM API 的开发者。一个完整的 Agent 系统通常还需要：

```text
业务目标
  ↓
任务定义与边界
  ↓
Agent / Workflow 设计
  ↓
Tool、数据和权限
  ↓
评估、观测和质量门禁
  ↓
上线、治理和持续改进
```

核心能力可以分成七类：

1. **任务建模：** 能否把模糊需求转成明确目标、输入、输出和边界。
2. **Agent 设计：** 能否判断什么时候用工作流、单 Agent 或多 Agent。
3. **工具与上下文：** 能否设计 Tool、Memory、Knowledge 和权限边界。
4. **SDD / Agentic SDLC：** 能否让 Agent 在结构化流程中工作。
5. **评估与可靠性：** 能否定义质量指标、回归测试和失败处理。
6. **治理与安全：** 能否控制数据、权限、风险、审计和责任。
7. **协作与交付：** 能否和产品、后端、测试、运营及其他 Agent 协作。

## 3. Agent 工程师专属评分模型

传统工程师的评分维度可以继续作为公司通用的观察语言，但 Agent 工程师需要一套更贴近岗位本质的最终能力模型。

这里建议采用“Agent 专属能力维度 + 传统维度作为证据标签”的方式：

- 最终评价回答候选人是否具备 Agent 工程能力。
- 记录时仍可以使用公司熟悉的 Testing、Development、Communication 等语言。
- 项目经历不作为一个孤立能力，而是作为所有能力维度的证据来源。

每个 Agent 专属维度使用内部统一的 1—5 级：

| 分数 | 英文等级 | 中文理解 |
|---:|---|---|
| 1 | Below Expectation | 低于预期：明显不足，无法满足岗位要求 |
| 2 | Basic | 基础：有初步了解，需要较多指导 |
| 3 | Competent | 胜任：能够独立完成岗位要求 |
| 4 | Proficient | 熟练：经验较完整，能够处理复杂情况 |
| 5 | Strong | 强：理解深入，能建立方法并影响他人 |

### 六个 Agent 专属能力维度

| Agent 能力维度 | 主要判断内容 | 可对应的传统观察维度 |
|---|---|---|
| Problem Framing & Agent Fit | 能否识别真正的问题，判断是否需要 Agent、Workflow、单 Agent 或多 Agent，并定义边界 | Problem-Solving Approach、Communication |
| Agent System Design & Orchestration | 能否设计 Agent 角色、状态、Tool、Context、Memory、协作协议和失败路径 | Development Capability、Problem-Solving Approach |
| Evaluation & Reliability | 能否定义任务成功标准，建立评估集、回归测试、Trace、质量门禁和恢复机制 | Testing Capability、Development Capability |
| Governance & Safe Autonomy | 能否设计权限、风险分级、人工审批、隐私、审计、回滚和责任边界 | Problem-Solving Approach、Project Experience |
| Delivery & Collaboration | 能否通过 SDD / Agentic SDLC 与产品、工程、测试、安全和运营协作，把系统交付上线 | Communication、Development Capability、Project Experience |
| Learning & Leverage | 能否从失败和线上反馈中学习，把经验沉淀为评估集、规范、平台能力和团队方法 | Learning Ability、Project Experience |

### 推荐的默认权重

如果岗位没有特殊要求，可以先使用以下权重：

| Agent 能力维度 | 权重 |
|---|---:|
| Problem Framing & Agent Fit | 15% |
| Agent System Design & Orchestration | 20% |
| Evaluation & Reliability | 20% |
| Governance & Safe Autonomy | 15% |
| Delivery & Collaboration | 15% |
| Learning & Leverage | 15% |

对于偏平台或生产稳定性的岗位，可以提高 `Evaluation & Reliability` 和 `Governance & Safe Autonomy`；对于偏探索和创新的岗位，可以提高 `Problem Framing & Agent Fit` 和 `Learning & Leverage`。

### 如何使用评分

- 每个维度给出 1—5 分，并记录具体证据，不要只写印象。
- 技术题答得好，不代表 Agent 能力都高；例如能讲框架但不会判断 Agent 是否适用，`Problem Framing & Agent Fit` 仍可能偏低。
- 项目经历不是“做过项目就高分”，而要看候选人能否从项目中提供决策、限制、指标和复盘证据。
- 评分时可以在证据旁标注传统观察标签，例如 `[Testing]`、`[Development]`、`[Communication]`，但最终填入六个 Agent 专属维度。
- 如果内部没有特别指定权重，建议使用上面的默认权重，而不是简单计算技术问题数量。
- 最终 Recommendation 不应只由平均分决定，还要看是否存在关键维度低于岗位底线。

### 单维度评分尺度

- **1 — Below Expectation：** 只会描述功能或框架，无法说明任务边界、验收标准、失败处理或风险控制。
- **2 — Basic：** 能在清晰方案和指导下完成单点工作，但缺少独立判断和系统性。
- **3 — Competent：** 能独立交付一个范围明确的 Agent 功能，并处理常见失败和评估问题。
- **4 — Proficient：** 能处理复杂任务、多 Agent 协作、线上问题和工程取舍，讲清指标与复盘。
- **5 — Strong：** 能建立可复用的 Agent 架构、评估体系、治理规范或团队交付方法，并影响组织。

# 第一部分：项目真实性与角色理解

## Q1. 请介绍一个你真正负责过的 Agent 或 LLM 应用项目

### 参考答案要点

完整回答应该包含：

- 业务目标和目标用户。
- Agent 具体负责什么，不负责什么。
- 系统输入、输出和主要流程。
- 候选人的个人贡献。
- 最大的技术或业务风险。
- 上线后的效果和衡量指标。
- 遇到过的失败、返工或降级。

### 追问

- 如果去掉你负责的部分，项目还能不能运行？
- 哪个决定是你做的，哪些决定是别人做的？
- 项目中最不满意的地方是什么？
- 如果现在重新做，你会改变什么？

### 低质量信号

- 只讲团队，不讲个人贡献。
- 只说用了哪些框架，不说解决了什么问题。
- 没有任何失败经验。
- 不能说出上线后的指标或用户反馈。

## Q2. Agent 工程师和普通后端工程师最大的区别是什么？

### 参考答案要点

较好的回答应该提到：

- Agent 的行为具有概率性，不完全是确定性逻辑。
- 需要设计任务边界、上下文、工具和决策流程。
- 不能只验证代码是否运行，还要验证输出质量和行为轨迹。
- 需要处理幻觉、错误调用、上下文污染、成本和模型变化。
- 需要建立评估、观测、回放和持续优化机制。

### 关键判断

候选人是否把 Agent 理解为“Prompt + API”，还是理解为一个需要交付和治理的系统。

## Q3. 你如何定义一个 Agent 项目是否成功？

### 参考答案要点

不能只看“模型回答得像不像人”。应同时考虑：

- 任务成功率。
- 正确性、完整性和事实一致性。
- 拒答或转人工是否正确。
- 用户满意度和业务转化。
- 成本、延迟和稳定性。
- 安全事件和违规率。
- 人工介入率。
- 维护和迭代成本。

# 第二部分：任务建模与 Agent 选择

## Q4. 什么情况下应该使用 Agent，而不是普通工作流？

### 参考答案要点

适合 Agent 的情况：

- 任务步骤不完全固定。
- 需要根据输入动态选择路径。
- 需要理解自然语言、非结构化内容或多个工具。
- 目标明确，但实现路径有多种可能。
- 需要根据中间结果进行下一步决策。

更适合普通工作流的情况：

- 规则稳定、步骤固定。
- 错误代价很高且必须确定性执行。
- 流程可以用明确规则表达。
- 需要强一致、可预测和容易审计。

### 低质量信号

“复杂问题就用 Agent，简单问题就用工作流”但无法解释复杂在哪里、风险如何控制。

## Q5. Agent 越自主越好吗？

### 参考答案要点

自主性应该按风险分级，而不是越高越好：

| 风险 | 示例 | 建议 |
|---|---|---|
| 低 | 总结、分类、草稿 | 可自动执行 |
| 中 | 生成代码、更新知识库 | 自动执行但需要 Review |
| 高 | 退款、删数据、发消息 | 必须人工确认 |
| 极高 | 医疗诊断、金融决策、权限变更 | 不应完全自动化 |

判断标准包括：错误代价、可逆性、可发现性、影响范围、责任和合规要求。

## Q6. 什么时候不应该做成 Agent？

### 参考答案要点

以下情况通常应该优先考虑确定性流程：

- 业务规则明确且变化少。
- 需要强一致和严格审计。
- 结果不能容忍随机性。
- 自动决策可能造成重大损失。
- Agent 带来的收益不足以抵消成本和复杂度。

成熟的 Agent 工程师不会为了使用 Agent 而使用 Agent。

## Q7. 如何把一个模糊需求转成 Agent 任务？

### 参考答案要点

至少要明确：

1. 谁是用户。
2. 用户想完成什么任务。
3. Agent 可以做什么。
4. Agent 明确不能做什么。
5. 可以访问哪些数据和工具。
6. 输出格式和质量标准是什么。
7. 什么情况下拒答、暂停或转人工。
8. 如何记录和评估结果。

# 第三部分：SDD 与 Agentic SDLC

## Q8. 你如何理解 SDD？

### 参考答案要点

SDD 可以理解为 Spec-Driven Development：把规格说明作为开发过程中的核心契约，先明确“做什么、为什么做、如何验收”，再拆解设计、任务和实现。

常见流程：

```text
Spec → Plan → Tasks → Implement → Validate → Review
```

好的 Spec 不一定很长，但至少应包含：

- 背景和目标。
- 用户或业务场景。
- 功能范围和非目标。
- 输入、输出和边界条件。
- 验收标准。
- 风险与约束。
- 需要人工决策的事项。

参考：[GitHub Spec Kit](https://github.github.com/spec-kit/index.html)、[What is SDD?](https://github.github.com/spec-kit/concepts/sdd.html)

## Q9. 如果产品经理说“做一个智能客服”，你会先问什么？

### 参考答案要点

- 服务哪些用户和渠道？
- 主要解决哪些问题？
- 哪些问题不在范围内？
- 是否允许查询订单、修改订单或退款？
- 什么情况下转人工？
- 正确率和响应时间要求是什么？
- 需要保存哪些会话和用户数据？
- 如何处理敏感信息？
- 上线后用什么指标判断成功？

## Q10. Agent 参与研发时，哪些阶段可以自动化？

### 参考答案要点

Agent 可以辅助：

- 需求澄清和问题整理。
- 生成 Spec 初稿。
- 方案对比和风险清单。
- 任务拆解。
- 代码实现。
- 测试生成和执行。
- 文档生成。
- 代码审查和安全检查。

但是否自动执行，取决于风险。高风险决策、架构基线、生产发布、权限变更和数据删除等动作通常需要人工批准。

## Q11. 如果 Spec、代码和测试不一致，应该怎么办？

### 参考答案要点

不能默认代码就是事实。应先判断：

- 是需求变更了？
- 是实现错误？
- 是测试过时了？
- 是 Spec 写得不完整？

确定后同步更新相关工件，并保留变更原因。否则会形成 Spec Drift，后续 Agent 会继续基于错误上下文工作。

## Q12. 你会如何为 Agent 设计质量门禁？

### 参考答案要点

可以设置分层门禁：

- Spec 完整性检查。
- 需求和任务的一致性检查。
- 代码和测试检查。
- 安全、隐私和敏感信息扫描。
- Agent 行为评估。
- 成本和延迟阈值。
- 人工 Review。
- 发布前灰度或回滚方案。

成熟做法是根据风险设置不同强度，而不是所有任务使用同一套流程。

# 第四部分：Tool、Context、Memory 与权限

## Q13. 如何设计一个 Tool？

### 参考答案要点

一个可被 Agent 使用的 Tool 应该有：

- 清晰的名称和用途。
- 明确的输入参数和类型。
- 输出结构和错误状态。
- 权限要求。
- 超时和重试策略。
- 幂等性说明。
- 副作用说明。
- 审计记录。

Tool 不应该只是把一个内部接口原样暴露给模型。

## Q14. Tool 的权限应该由谁判断？

### 参考答案要点

不能让模型自己决定用户有没有权限。应由系统根据：

- 用户身份。
- 业务角色。
- 资源归属。
- 操作类型。
- 风险等级。
- 当前审批状态。

进行权限判断。模型可以提出调用请求，但最终授权必须由确定性系统完成。

## Q15. 如何管理 Agent 的上下文？

### 参考答案要点

需要区分：

- 当前任务上下文。
- 对话短期上下文。
- 长期用户记忆。
- 结构化业务数据。
- 外部知识和证据。
- Agent 之间的中间状态。

不应该把所有历史内容一股脑塞给模型。需要考虑相关性、隐私、成本、延迟和上下文污染。

## Q16. 长期记忆应该保存什么？

### 参考答案要点

适合保存：

- 用户明确表达且长期有效的偏好。
- 用户授权保存的稳定事实。
- 经过确认的历史摘要。

不应该直接保存：

- 模型未经确认的推断。
- 不必要的敏感信息。
- 无期限的全部聊天记录。
- 无法追溯来源的摘要。

长期记忆应支持查看、修改、删除、过期和权限控制。

## Q17. 如果 Agent 的上下文越来越长，你会怎么处理？

### 参考答案要点

- 使用任务摘要。
- 只检索相关历史。
- 将稳定事实结构化存储。
- 清理重复和过期信息。
- 分离用户可见上下文与内部状态。
- 给不同类型信息设置优先级和生命周期。

## Q18. 如何防止 Prompt Injection 影响 Tool 调用？

### 参考答案要点

- 不把外部文本视为可信指令。
- 系统指令、用户输入、检索内容和 Tool 结果分层处理。
- Tool 参数进行 Schema 校验。
- 由业务系统进行权限校验。
- 对高风险操作要求人工确认。
- 限制 Tool 白名单、调用次数和递归深度。
- 记录和审计可疑调用。

# 第五部分：多 Agent 协作

## Q19. 什么情况下应该使用多 Agent，而不是单 Agent？

### 参考答案要点

适合拆分的情况：

- 有清晰的专业职责边界。
- 不同角色需要不同上下文或权限。
- 任务可以并行处理。
- 需要独立审查或交叉验证。
- 不同阶段的目标和输出格式不同。

不适合拆分的情况：

- 只是为了显得架构复杂。
- Agent 之间没有清晰职责。
- 任务本身很简单。
- 通信和调度成本超过收益。
- 多个 Agent 会共享全部权限和全部上下文。

## Q20. 如何拆分多 Agent 的职责？

### 参考答案要点

每个 Agent 都应有：

- 单一或相对集中的职责。
- 明确的输入。
- 明确的输出。
- 可用工具列表。
- 禁止事项。
- 成功标准。
- 失败处理方式。

拆分依据应是职责、上下文和权限边界，而不是“一个 Agent 对应一个技术名词”。

## Q21. 多 Agent 之间如何通信？

### 参考答案要点

优先使用结构化协议，而不是随意传递长段自然语言。协议应描述：

- 任务 ID。
- 当前阶段。
- 输入数据。
- 证据或来源。
- 结论。
- 置信度或不确定性。
- 未解决问题。
- 错误状态。
- 下一步建议。

同时应记录每次交互，以便回放和定位问题。

## Q22. 多 Agent 结论冲突时如何处理？

### 参考答案要点

不能简单采用多数投票。应考虑：

- Agent 的职责权威性。
- 证据来源和新鲜度。
- 风险等级。
- 是否存在安全 Agent 的否决权。
- 是否需要再次验证。
- 是否应该转人工。

高风险场景通常采用保守策略：无法确认时不执行，并明确说明不确定性。

## Q23. 如何设计多 Agent 的上下文共享？

### 参考答案要点

- 共享必要信息，不共享所有信息。
- 用户隐私数据按权限过滤。
- 中间结果结构化。
- 区分事实、推断和建议。
- 保留来源和证据。
- 控制上下文大小。
- 避免一个错误结论被所有 Agent 继承。

## Q24. 一个 Agent 失败后，系统应该怎么办？

### 参考答案要点

先区分失败类型：

- 输入不完整。
- Tool 暂时不可用。
- 输出格式错误。
- 逻辑判断不确定。
- 超时或成本超限。
- 权限或安全检查失败。

不同失败类型可以采取：

- 补充信息。
- 有限重试。
- 切换备用 Tool 或模型。
- 降级到确定性流程。
- 转人工。
- 中止并保留失败记录。

不能用无限重试掩盖问题。

## Q25. 如何评估多 Agent 系统，而不是只看最终答案？

### 参考答案要点

既要看最终任务成功率，也要看：

- 路由是否正确。
- Agent 是否调用了正确的 Tool。
- 是否出现越权调用。
- 中间结果是否符合协议。
- 是否发生无效循环。
- 是否使用了不必要的 Agent。
- 成本和延迟是否可接受。
- 失败是否能被发现和恢复。

## Q26. 如何证明多 Agent 比单 Agent 更好？

### 参考答案要点

需要进行对照实验，比较：

- 任务成功率。
- 事实正确性。
- 复杂任务完成率。
- 人工介入率。
- 延迟。
- Token 和调用成本。
- 失败恢复能力。
- 可解释性和可维护性。

如果多 Agent 只增加成本和复杂度，没有明显业务收益，就应该回退到单 Agent 或普通工作流。

# 第六部分：评估、可靠性与线上运营

## Q27. 如何评估一个 Agent 的质量？

### 参考答案要点

至少分三层：

**任务层：** 是否完成目标、是否满足验收条件。
**行为层：** 是否选择了正确路径、工具和参数。
**系统层：** 成本、延迟、稳定性、人工介入和安全事件。

还要针对不同业务建立 golden dataset 或回归案例，而不是只凭主观感受。

## Q28. 你怎么看 LLM-as-a-Judge？

### 参考答案要点

优点：

- 可批量评估开放式答案。
- 能辅助评价风格、完整性和相关性。
- 比全部依赖人工更高效。

局限：

- Judge 本身也可能犯错。
- 可能偏好某种表达风格。
- 可能受答案长度、措辞和模型家族影响。
- 不能替代人工和确定性规则。
- 关键结论需要人工抽检和校准。

## Q29. 如何建立 Agent 回归测试集？

### 参考答案要点

测试集应覆盖：

- 正常问题。
- 边界问题。
- 无法回答的问题。
- 恶意输入和 Prompt Injection。
- 工具失败。
- 信息冲突。
- 多轮上下文。
- 高风险操作。
- 多语言和不同表达方式。

每次线上发现一个有价值的问题，都应考虑是否加入回归集。

## Q30. Agent 上线后用户反馈变差，你如何定位？

### 参考答案要点

按链路拆查：

```text
用户输入
→ 任务识别
→ 上下文和记忆
→ Tool / 知识检索
→ Agent 决策
→ 最终生成
→ 安全审核
→ 用户看到的结果
```

要能区分：需求问题、输入问题、检索问题、工具问题、模型问题、Prompt 问题、汇总问题和评估问题。

## Q31. 如何观察 Agent 的线上行为？

### 参考答案要点

至少记录：

- 请求和任务 ID。
- Agent、Prompt 和模型版本。
- 输入输出摘要。
- Tool 调用和参数。
- 每一步耗时。
- Token、费用和重试次数。
- 最终结果和人工介入。
- 用户反馈。

敏感数据不能为了观测而无边界地记录，需要脱敏、访问控制和保留期限。

## Q32. 如何控制 Agent 的成本和延迟？

### 参考答案要点

- 避免不必要的 Agent 和 Tool 调用。
- 按任务难度做模型路由。
- 控制上下文长度。
- 对稳定结果进行缓存。
- 对并行任务设置上限。
- 使用超时、重试、熔断和降级。
- 建立按用户、团队、任务和模型的成本指标。
- 对高成本任务设置预算和人工确认。

## Q33. 如果模型供应商突然不可用，你会怎么办？

### 参考答案要点

- 通过统一网关隔离供应商差异。
- 配置备用模型或备用供应商。
- 对不同任务设置降级策略。
- 对不可降级的任务转人工或稍后重试。
- 设置超时、熔断和限流。
- 记录故障期间的影响范围。
- 恢复后进行数据补偿和复盘。

# 第七部分：AI 治理、安全与责任

## Q34. AI 治理至少要治理什么？

### 参考答案要点

- 数据和隐私。
- 模型和供应商。
- Prompt、知识库和版本。
- Tool 和权限。
- 输出安全。
- 高风险决策。
- 成本和资源。
- 日志、审计和追责。
- 评估和持续监控。
- 事故响应和回滚。

治理不是额外文档，而是把风险控制嵌入设计、开发、测试、发布和运营流程。

## Q35. 如果 Agent 可以退款、删除数据或发送外部消息，你会如何治理？

### 参考答案要点

- 读操作和写操作分级。
- 高风险操作需要人工确认。
- 用户身份和业务权限由系统校验。
- Tool 参数进行严格验证。
- 限制调用次数、范围和金额。
- 记录完整审计日志。
- 支持撤销、补偿或回滚。
- 发生异常时暂停自动化。

## Q36. 如何处理敏感数据和隐私？

### 参考答案要点

- 数据最小化。
- 明确采集和使用目的。
- 脱敏或匿名化。
- 权限分层。
- 传输和存储加密。
- 控制发送到外部模型的数据。
- 设定保留和删除策略。
- 记录访问审计。
- 允许用户查看、更正和删除可管理的记忆。

## Q37. 如果 Agent 输出了有害或错误内容，你会怎么办？

### 参考答案要点

短期：

- 阻断或下线有问题的流程。
- 保留证据和影响范围。
- 对用户做适当提示或人工接管。
- 修复 Prompt、Tool、数据或规则。

长期：

- 将案例加入评估集。
- 增加安全规则和边界测试。
- 重新评估模型、Workflow 和权限。
- 完成复盘并明确责任。
- 通过灰度和监控重新上线。

## Q38. AI 系统出了问题，责任应该由谁承担？

### 参考答案要点

不能把责任归因于“模型自己说的”。责任应该通过系统设计明确：

- 业务负责人负责目标和风险接受。
- Agent 工程师负责系统边界和技术控制。
- Tool 所属团队负责接口和业务权限。
- 安全、合规或法务负责相应审查。
- 人工审批者对自己批准的高风险动作负责。
- 系统应保留足够审计记录支持复盘。

好的治理设计应该让责任可追踪，而不是出了问题后再寻找替罪羊。

## Q39. 如何防止 Prompt Injection？

### 参考答案要点

- 将外部内容视为数据，不视为可信指令。
- 分离系统指令、用户输入、检索内容和 Tool 结果。
- Tool 参数和权限由确定性代码校验。
- 对敏感操作要求二次确认。
- 限制可用 Tool 和调用范围。
- 对高风险输入做检测和隔离。
- 记录异常行为并进行红队测试。

## Q40. 如何管理模型、Prompt、Tool 和知识库的变更？

### 参考答案要点

- 全部版本化。
- 保留变更原因和审批记录。
- 上线前跑固定评估集。
- 先灰度，再扩大流量。
- 关键指标回退时自动或人工回滚。
- 记录一次回答使用了哪些版本。

# 第八部分：协作、交付与团队能力

## Q41. 如何和产品经理一起定义 Agent 的验收标准？

### 参考答案要点

验收标准不能只写“回答准确”，应该包含可观察行为，例如：

- 对已知问题给出正确答案。
- 对未知问题明确拒答。
- 特定条件下必须调用某个 Tool。
- 高风险场景必须转人工。
- 不得泄露某类数据。
- 响应时间和成本不超过阈值。
- 失败时给出可理解的提示。

## Q42. 产品希望快速上线，但你认为风险太高，怎么办？

### 参考答案要点

- 先明确风险是什么以及可能影响谁。
- 用数据或测试结果说明风险，而不是泛泛说“AI 不可靠”。
- 缩小范围或降低自动化程度。
- 增加人工审核和灰度流量。
- 设定上线门槛、监控和回滚条件。
- 形成业务方确认的风险记录。

## Q43. 如何把 Agent 经验沉淀成团队规范？

### 参考答案要点

可以沉淀为：

- Agent 设计模板。
- Tool 设计规范。
- Spec 和验收标准模板。
- 多 Agent 通信协议。
- 评估集和回归测试规范。
- 权限和治理检查清单。
- 线上事故复盘模板。
- 版本与发布流程。

## Q44. 你带新人做 Agent 项目时，会要求他先做什么？

### 参考答案要点

不是马上写 Prompt，而是要求他先：

1. 说明业务目标。
2. 画出任务流程。
3. 写清 Agent 边界和工具权限。
4. 定义几个成功和失败案例。
5. 明确评估方式。
6. 再做最小可行版本。

## Q45. 你如何判断一个 Agent 项目已经可以交付？

### 参考答案要点

- 需求和非目标清楚。
- 主要路径通过评估。
- 失败路径有处理方式。
- 高风险动作有人工控制。
- 输入输出和 Tool 协议稳定。
- 关键日志和指标可用。
- 成本和延迟可接受。
- 有灰度、回滚和事故响应方案。
- 团队能够维护，而不是只有原作者知道怎么运行。

# 第九部分：通用场景题

## 场景一：企业知识助手

> 设计一个面向公司员工的知识助手。它可以回答制度问题、查询业务数据、生成报告，必要时创建工单。

### 观察维度

- 如何定义不同任务的边界？
- 哪些内容可以直接回答？
- 哪些操作必须确认？
- 如何处理不同部门的数据权限？
- 如何保证回答有来源？
- 如何处理知识冲突和过期？
- 如何评估回答质量？

## 场景二：研发协作 Agent

> 设计一个参与软件研发的 Agent 团队，从需求分析到代码发布都可以提供帮助。

### 观察维度

- 是否使用 Spec、Plan、Tasks 等中间工件？
- Agent 角色如何拆分？
- 如何防止多个 Agent 重复或互相覆盖？
- 谁负责最终审批？
- 如何做代码、测试、安全和文档检查？
- 如何记录 Agent 的决策和变更？

## 场景三：高风险业务 Agent

> 设计一个可以分析用户资料并给出建议的 Agent，错误可能造成较大业务或用户风险。

### 观察维度

- 是否会主动降低自动化范围？
- 是否会要求人工确认？
- 是否考虑数据隐私、审计和解释？
- 如何处理不确定性和冲突信息？
- 是否有回滚、申诉和事故响应机制？

# 第十部分：不同级别的区别

## 初级 Agent 工程师

应具备：

- 能调用模型和工具完成明确任务。
- 能理解 Prompt、上下文和结构化输出。
- 能写基本的失败处理和日志。
- 能根据既定评估集验证结果。
- 知道敏感数据和高风险工具不能随意开放。

## 中级 Agent 工程师

应具备：

- 能独立进行任务建模和 Agent / Workflow 选择。
- 能设计 Tool、Context、Memory 和权限边界。
- 能建立评估集、回归测试和线上观测。
- 能处理多 Agent 协作、冲突和失败恢复。
- 能平衡质量、成本、延迟和维护性。

## 高级 Agent 工程师 / Agent Tech Lead

应具备：

- 能建立团队级 Agent 架构和交付规范。
- 能设计 SDD / Agentic SDLC。
- 能建立质量门禁、治理机制和责任边界。
- 能判断哪些能力应该平台化，哪些应保持业务自治。
- 能推动产品、工程、安全、合规和运营形成共同标准。
- 能从线上反馈建立持续改进闭环。

## 第十一部分：面试红旗清单

出现以下情况时，应继续追问，不要只根据表达流畅度判断：

- 把 Agent 等同于 Prompt + Tool Calling。
- 只会介绍框架，不会讲业务目标和用户价值。
- 认为“多 Agent”就是多创建几个角色。
- 不知道 Agent 之间应该传递什么结构化信息。
- 多 Agent 冲突时只说“投票”。
- 认为 Agent 输出正确就不需要评估和回放。
- 不考虑权限，默认 Agent 可以访问所有数据和工具。
- 认为所有高风险动作都可以自动化。
- 没有失败、返工、灰度、降级或回滚经验。
- 不能区分模型问题、数据问题、Tool 问题和流程问题。
- 只讲准确率，不讲成本、延迟、稳定性和责任。
- 不知道如何把线上问题加入评估集。
